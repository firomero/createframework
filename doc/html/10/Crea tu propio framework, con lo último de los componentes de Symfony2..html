Crea tu propio framework, con lo último de los componentes de Symfony2.

Felix Ivan Romero Rodríguez Febrero 23, 2015 Arquitectura y diseño, Desarrollo web, DeSymfony, Patrones, PHP, Symfony No comments

En esta ocasión traemos una de las potencialidades del componente HttpKernel, y nos referimos a la cache de la aplicación. Elemento importante a considerar en cuanto al rendimiento de nuestras soluciones. Sin más Capítulo 10, Caché

Entre las facilidades que conlleva la utilización de los componentes de Symfony2 está la interoperabilidad entre las aplicaciones que lo utilicen. Como ejemplo de componentes que permiten este comportamiento está el HttpKernel. Cumpliendo este objetivo, implementemos la interface HttpKernelInterface. Esta interface es la pieza clave del componente HttpKernel. Sabiendo esto actualicemos el comportamiento de nuestro framework .


<?php

// example.com/src/Framework.php

// ...

use Symfony\Component\HttpKernel\HttpKernelInterface;

class Framework implements HttpKernelInterface
{
// ...

public function handle(Request $request, $type = HttpKernelInterface::MASTER_REQUEST, $catch = true)
{
// ...
}
}

Por sencillo que parezca, conlleva mucho. Situemos por ejemplo la situación del cacheo HTTP de nuestro sistema. El sistema de cache implementa un proxy reverso, escrito completamente en PHP, encapsulador en una instancia de HttpKernel.

// example.com/web/front.php

use Symfony\Component\HttpKernel\HttpCache\HttpCache;
use Symfony\Component\HttpKernel\HttpCache\Store;

$framework = new Simplex\Framework($dispatcher, $matcher, $resolver);
$framework = new HttpCache($framework, new Store(__DIR__.'/../cache'));

$framework->handle($request)->send();

Y eso es todo lo que toma incorporar soporte de cache HTTP. Configurar la cache HTTP. Configurar la cache es necesario realizarlo a través de las cabeceras de cache. Por ejemplo para cachear una respuesta por 10 segundos se utilizaría Response::setTtl()


public function indexAction(Request $request, $year)

{
$leapyear = new LeapYear();
if ($leapyear->isLeapYear($year)) {
$response = new Response('Yep, this is a leap year!');
} else {
$response = new Response('Nope, this is not a leap year.');
}

$response->setTtl(10);

return $response;
}

Al desplegar aplicaciones web, es aconsejable mantener deplegado proxys reversos con el componente de Symfony2, o aun mejor utilizar proxys de mayor eficiencia como Varnish. Un objeto Response contiene objetos que permiten configurar la cache, encontramos en ese conjunto setCache, que de manera abstracta, permite configurar la Cache en un simple arreglo

$date = date_create_from_format('Y-m-d H:i:s', '2005-10-15 10:00:00');

$response->setCache(array(
'public'        => true,
'etag'          => 'abcde',
'last_modified' => $date,
'max_age'       => 10,
's_maxage'      => 10,
));
Lo cual es equivalente:
$response->setPublic();
$response->setEtag('abcde');
$response->setLastModified($date);
$response->setMaxAge(10);
$response->setSharedMaxAge(10);
Cuando se utiliza un modelo de validación el método isNotModified, perm,ite rápidamente en tiempo de respuesta cortocircuitar la generación de la respuesta
$response->setETag('whatever_you_compute_as_an_etag');

if ($response->isNotModified($request)) {
return $response;
}
$response->setContent('The computed content of the response');

return $response;

Hasta este punto nos hemos puesto al día de lo que significa cachear la aplicación, pero que pasa cuando se quiere cachear solo algunas partes de la aplicación y no la página completa que se genera. Es entonces cuando en acción vienen las ESI(Edge Side Includes). En lugar de generar todo un contenido se puede configurar que parte se quiere ir generando. Para que las ESI funcione, debe existir un proxy reverso que logre interpretarlas, tales como la implementación que propone Symfony2 o mejor, utilizar Varnish.
